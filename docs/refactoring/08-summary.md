## Приоритеты по urgency/impact

### 🔴 P0 — Критические (баги)

Исправить немедленно. Могут вызывать некорректную работу.

| # | Проблема | Файл | Риск |

| 1 | Отсутствие cycle guard в parent chain | `FileService::canAccess()`, `SharedFolderController::canAccess()` | Бесконечный цикл при зацикленной структуре папок |

## Отложенные позиции

Эти пункты исключены из текущего плана рефакторинга — они либо дают низкий ROI, либо требуют архитектурных изменений вне рамок рефакторинга.

| Позиция | Почему отложено | Когда вернуться |
|---------|----------------|-----------------|
| Разбивка `folders-tree` (24-32ч) | Высокий риск регрессий, требует стабильной фазы | После закрытия всех P0-P1 |
| `computeAvailableTypeGroups` мемоизация | Имеет смысл только при больших объёмах данных | При росте БД > 100k файлов |
| `share-target.component.ts` inline → файлы | Чисто косметика, нулевой функциональный эффект | При плановом рефакторинге шаблонов |
| ETag консистентность | Реально не решается для multipart S3 uploads без изменения архитектуры upload | Версия 2.0 |
| `editorExpanded` sync с query param | Неочевидно нужно — пользователь редко перезагружает страницу с открытым редактором | По user feedback |

## План работ по этапам

### ### Этап 1.1: Критические баги (P0) — ~3ч

1. Добавить cycle guard в `FileService::canAccess()` и `SharedFolderController::canAccess()`
2. Добавить проверку статуса в `completeUpload()`
3. Исправить `checkStorageQuota()` — учитывать статус и версии
4. Убрать лишние link-поля из `formatFileCard()` SharedFolderController
5. Добавить очистку S3 при cancelUpload

### Этап 1.2: Централизация сервисов (P1-бэкенд) — ~10ч

1. Создать `S3UrlService` — перенести всю генерацию URL
2. Создать `MimeService` — перенести всю классификацию MIME
3. Создать `FileCardBuilder` / единый метод `toCard()` + синхронизировать `buildListItem()`
4. Вынести access checks в единое место
5. Вынести quota checks в единое место
6. Оптимизировать `SharedFolderController::index()` — рекурсивный запрос (с учётом версии БД)
7. Исправить N+1 в `formatFolder()`

### Этап 1.3: Оптимизация запросов (P1-P2-бэкенд) — ~6ч

1. Мемоизировать `computeAvailableTypeGroups()`
2. Объединить запросы в `FileController::show()` (опциональный eager load)
3. Убрать fallback-запрос из `buildListItem()`
4. Оптимизировать `File::canBeDeleted()`
5. Унифицировать TTL для S3 presigned URL

### Этап 2.1: Shared утилиты (P1-фронтенд) — ~3ч

1. Создать `@shared/utils/format.ts` — `formatSize()`
2. Создать `@shared/utils/file.ts` — `canViewInBrowser()`, `classifyMimeType()`
3. Создать `@shared/utils/tree.ts` — `flattenTree()`
4. Создать `@shared/constants/limits.ts` — `PLAN_FILE_LIMITS`
5. Заменить везде импорты на shared

### Этап 2.2: Централизация upload (P1-фронтенд) — ~4ч

1. Расширить `FileUploadService.upload()` — поддержка shared folder
2. Убрать `uploadToSharedFolder()` из `folders-tree.component.ts`
3. Убрать дублирование в `shared-folders.component.ts`
4. Использовать `FileTypeIconComponent` везде

### Этап 2.3: MD-редактор рефакторинг (P1-фронтенд) — ~6ч

1. Создать `MarkdownEditorService` (сервис с DI, не базовый класс)
2. Вынести общую логику создания TipTap редактора
3. Вынести save/revert/lock логику
4. Оставить два компонента только для UI-шаблонов

### Этап 2.4: Разбивка больших компонентов (P2-фронтенд) — ~24-32ч ⚠️

**Наиболее рисковый рефакторинг.** Неправильная декомпозиция вызовет баги в drag-drop, мультиселекте, состоянии диалогов. Требует стабильной фазы без новых фич.

1. Разделить `folders-tree.component` на 5+ под-компонентов
2. Разделить `file-detail.component` на 6+ под-компонентов
3. Выделить `admin.component` и `support.component` — создать `SupportTicketService`
