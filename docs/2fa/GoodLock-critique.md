# Критика ТЗ GoodLock

> Дата: 2026-06-11
> Исходный документ: `GoodLock.md`

## 1. Критичные пробелы (без них невозможно начать разработку)

### 1.1. API — форматы запросов/ответов

Есть только URL-методы (`POST /api/2fa/session/create`), но нет JSON-схем:
какие поля отправлять, какие получать, какие типы данных. Нужен OpenAPI
(spec) или хотя бы примеры payload для каждого метода. Без этого два
разработчика неизбежно напишут несовместимые имплементации.

### 1.2. Аутентификация в мобильном приложении

ТЗ описывает только 2FA для web-входа, но мобильное приложение тоже
должно авторизоваться на сервере. Не указано:
- JWT? Sanctum token? OAuth2?
- Какой flow при первом запуске / регистрации?
- Хранится ли токен между сессиями, как обновляется?

### 1.3. Связь с основным приложением (Delifile)

GoodLock — это отдельный сервис или модуль внутри Delifile?
- Если отдельный: как синхронизируются пользователи? SSO? Общая БД?
- Если модуль: какие endpoints Delifile используются, как происходит
  интеграция с существующей авторизацией?
Без ответа архитектура приложения не определена.

### 1.4. TOTP — детали реализации

Указано «одноразовый код (TOTP)», но не указано:
- Алгоритм: HMAC-SHA1? SHA256? SHA512?
- Период: 30 сек (стандарт) или иной?
- Количество цифр: 6 или 8?
- Как секрет генерируется и передаётся при привязке устройства?
- Какой формат QR для привязки TOTP (otpauth:// URI)?
Без этих параметров невозможно реализовать TOTP-генератор в приложении.

### 1.5. UX/UI — полное отсутствие

Нет ни одного макета, нет дизайна, нет цветовой палитры, нет гайдлайнов.
Для мобильного приложения это критично: без дизайна разработчик будет
принимать решения на ходу, что приведёт к несовместимому UX между
 платформами иускорит накопление технического долга.

## 2. Важные пробелы (замедлят разработку)

### 2.1. Apple Developer Account setup

В ТЗ есть настройка Mac-раннера и EAS CLI, но нет шагов по настройке
Apple Developer Portal:
- Создание App ID с нужными capabilities (Push Notifications, Keychain)
- Создание Distribution Certificate
- Создание Provisioning Profile
- Настройка App Store Connect записи приложения
- Добавление APNs Key (.p8) в Firebase

EAS не создаёт сертификаты автоматически для production.

### 2.2. Firebase Project setup

Нет инструкции по настройке Firebase:
- Создание проекта в Firebase Console
- Включение Cloud Messaging
- Скачивание `google-services.json` (Android) и `GoogleService-Info.plist` (iOS)
- Добавление APNs Authentication Key в Firebase Console
- Регистрация приложений в Firebase

### 2.3. Локальная разработка

Нет Docker Compose или иных инструкций по поднятию dev-окружения с нуля.
Как разработчику запустить.backend + фронтенд + БД локально для отладки?
Без этого onboarding нового разработчика занимает дни.

### 2.4. Тестирование

Ни слова о тестировании: unit, integration, e2e. Критерии приёмки
требуют «работает», но без набора тестов непонятно, как валидировать
регрессии при последующих изменениях. Нужен хотя бы минимальный план:
- PHPUnit для backend
- Jest для Angular
- Detox / Maestro для мобильного E2E

### 2.5. Фоновая обработка push на iOS

iOS имеет жёсткие ограничения на background processing. Нужен конкретный
план: silent push vs. notification extension, что делать если app killed.
Не указано, как обрабатывать push когда приложение:
- В foreground — понятно
- В background — Content-Available notification?
- Killed — только notification tap, без background fetch

## 3. Мелкие уточнения

### 3.1. Rate limiting

Упомянут HTTP 429, но нет конкретных лимитов. Сколько попыток ввода
кода в минуту? Сколько запросов на создание сессии? Без этого нельзя
настроить throttle и нельзя протестировать.

### 3.2. SSE vs Polling — выбор

ТЗ пишет «polling или SSE», но не выбирает. Для production нужно
определиться: SSE — лучше для UX, но сложнее в инфраструктуре
(reverse proxy buffering). Polling — проще, но нагружает сервер.
Нужно решение и его обоснование.

### 3.3. Мониторинг и эксплуатация

Не упомянуты: Sentry, лог-ротация, health checks, uptime monitoring.
Для production-приложения это обязательно.

### 3.4. Масштабирование

Нет раздела о горизонтальном масштабировании. Что происходит при
тысяче одновременных 2FA-сессий? Queue workers — сколько? Redis или
database для очередей при росте нагрузки?

### 3.5. Резервное копирование БД

Не указана стратегия бэкапов для MySQL. Автоматический бэкап?
Ручной? Куда?

## 4. Что хорошо

- Детальная структура БД и статусы — сразу можно писать миграции.
- Раздел инфраструктуры (Часть II) — отличный, с готовыми скриптами и
  конфигами. Это редкость в ТЗ.
- CI/CD описан полностью, включая macOS runner — это готов к применению
  процесс.
- Push-уведомления включены в ТЗ (в отличие от исходного ТЗ, где их не
  было — автор сам это зафиксировал).
