# ТЗ / спецификация v2: комментарии к файлам и папкам в DeliFile

## Обзор

DeliFile уже содержит базовые сущности, на которые должен опираться модуль комментариев: проводник `/folders` с разделением на локальные и общие папки, карточку файла `/files/:id`, список пользователей с доступом к файлу, активность по файлу, версионирование файлов и механизм общих папок с ролями доступа.[cite:1] При этом один и тот же файл может быть доступен нескольким пользователям и может быть добавлен участником общей папки в его собственные «Мои файлы», то есть пользовательская структура папок не является общей для всех участников работы с файлом.[cite:1]

Из этого следует ключевой архитектурный принцип: shared-комментарии должны привязываться к объекту файла или общей папки, а не к локальному размещению файла в персональной структуре пользователя.[cite:1] Локальные папки не являются совместно используемыми сущностями в текущей модели продукта, поэтому для них в рамках MVP поддерживаются только личные комментарии пользователя.[cite:1]

## Цели MVP

MVP должен дать пользователю возможность:

- оставлять общие комментарии к файлам;
- оставлять общие комментарии к общим папкам;
- оставлять личные комментарии к файлам, общим папкам и локальным папкам;
- отключать shared-комментарии на уровне файла;
- задавать политику shared-комментариев на уровне общей папки через `shared_comments_mode`;
- видеть единый shared-thread файла независимо от того, в какой локальной папке у конкретного пользователя находится этот файл.[cite:1]

В первую версию не входят shared-комментарии к локальным папкам, потому что локальные папки принадлежат одному пользователю и не шарятся между участниками, а значит не образуют общую предметную сущность для совместного обсуждения.[cite:1]

## Базовые продуктовые решения

### Что считается целевой сущностью

В рамках MVP комментарии поддерживаются для следующих target types:

| Сущность | Shared comments | Private comments |
|---|---|---|
| `file` | Да | Да |
| `shared_folder` | Да | Да |
| `local_folder` | Нет | Да |

### Что не является target для shared-треда

Shared-тред не может быть привязан к локальному размещению файла в дереве папок пользователя. Это важно, потому что DeliFile уже позволяет участнику общей папки добавить файл в «Мои файлы», и после этого один и тот же файл может находиться в разных локальных папках у разных пользователей.[cite:1] Следовательно, если файл обсуждается совместно, обсуждение должно быть единым на уровне `file_id`, а не зависеть от `folder_id` у конкретного пользователя.[cite:1]

### Разделение типов комментариев

- `shared` — комментарии, видимые всем пользователям, имеющим доступ к объекту.[cite:1]
- `private` — комментарии, видимые только автору; они не зависят от shared-настроек и всегда доступны пользователю в рамках его собственного доступа к объекту.[cite:1]

## Правила предметной области

### 1. Файл

Файл — самостоятельная сущность с карточкой, действиями, правами доступа, списком пользователей, активностью и версионированием.[cite:1] Поэтому:

- shared-комментарии к файлу существуют как единый тред на уровне `file_id`;
- private-комментарии к файлу существуют как личный тред пользователя на уровне `file_id`;
- размещение файла в локальной папке пользователя не влияет на shared-thread файла.[cite:1]

### 2. Общая папка

Общая папка — совместная сущность с отдельным управлением доступом, ролями «Просмотр» и «Редактирование», загрузкой файлов и публичной ссылкой на папку.[cite:1] Поэтому:

- общая папка может иметь собственный shared-thread;
- общая папка может иметь private-thread у каждого пользователя;
- на уровне общей папки вводится политика `shared_comments_mode`.

### 3. Локальная папка

Локальная папка существует только в персональной структуре пользователя внутри `/folders`.[cite:1] В текущей модели продукта она не шарится другим пользователям и не является общей сущностью. Поэтому:

- shared-комментарии для `local_folder` отсутствуют;
- допускаются только private-комментарии автора;
- если в будущем появится шаринг локальных папок, это потребует отдельной ревизии модели target types.

## Политика `shared_comments_mode`

### Общая папка

Для `shared_folder` в MVP вводится поле:

- `shared_comments_mode = enabled | disabled | inherit_for_items`

Смысл режимов:

| Режим | Комментарии к общей папке | Комментарии к файлам внутри общей папки |
|---|---|---|
| `enabled` | Разрешены | По умолчанию разрешены |
| `disabled` | Запрещены | По умолчанию запрещены |
| `inherit_for_items` | Разрешены для самой папки | Файлы внутри папки наследуют правило папки, если на файле нет override |

Для практической реализации MVP рекомендуется трактовать `inherit_for_items` так: shared-комментарии к самой общей папке включены, а для файлов внутри папки действует логика наследования через file-level override. Это делает режим полезным и предсказуемым, а не дублирующим `enabled`.

### Файл внутри общей папки

Чтобы файл мог переопределить правило общей папки, вводится поле:

- `shared_comments_override = inherit | enabled | disabled`

Итоговое право на shared-комментарий к файлу считается так:

1. Если файл открыт вне контекста общей папки, действует `file.shared_comments_enabled`.
2. Если файл открыт в контексте общей папки:
   - при `override = enabled` shared-комментарии разрешены;
   - при `override = disabled` shared-комментарии запрещены;
   - при `override = inherit` применяется `shared_folder.shared_comments_mode`.

### Файл вне общей папки

Если файл не рассматривается как элемент общей папки, используется отдельный file-level флаг:

- `shared_comments_enabled: boolean`

Это покрывает личные файлы, напрямую расшаренные файлы и любые сценарии, где файл не зависит от политики общей папки.[cite:1]

## Права доступа

Текущий продукт уже содержит владельца файла, получателя файла, сохранённый файл у пользователя, а также роли в общей папке «Просмотр» и «Редактирование».[cite:1] На этой основе права на комментарии рассчитываются не только по роли, но и по effective policy объекта.

### Основная формула

Для shared-комментария пользователь может писать комментарий только если одновременно выполняются условия:

- у пользователя есть доступ к объекту;
- effective shared comments policy разрешает комментирование;
- у пользователя есть право `can_comment`.

Private-комментарии не зависят от shared policy и доступны пользователю, если он имеет право видеть объект.

### Матрица прав MVP

| Контекст | Видеть объект | Видеть shared | Писать shared | Видеть private | Писать private |
|---|---|---|---|---|---|
| Владелец файла | Да [cite:1] | Да | Да, если политика разрешает | Да | Да |
| Получатель файла | Да [cite:1] | Да | Да, если политика разрешает и есть `can_comment` | Только свои | Да |
| Участник общей папки `Просмотр` | Да [cite:1] | Да | Да, если политика разрешает и есть `can_comment` | Только свои | Да |
| Участник общей папки `Редактирование` | Да [cite:1] | Да | Да, если политика разрешает | Только свои | Да |
| Локальная папка владельца | Да [cite:1] | Нет | Нет | Да | Да |
| Пользователь по публичной ссылке | Да в рамках ссылки [cite:1] | Нет | Нет | Нет | Нет |

### Право изменения настроек

Изменять shared comment policy может:

- владелец файла — для файла;[cite:1]
- владелец / создатель общей папки или пользователь с правом управления доступом — для общей папки;[cite:1]
- владелец локальной папки может управлять только private UI для своих заметок, но shared policy для local folder отсутствует.

## Effective policy rules

### Файл

#### Сценарий A: файл вне общей папки

Используется:

- доступ к файлу;
- `file_comment_settings.shared_comments_enabled`.

#### Сценарий B: файл в контексте общей папки

Используется:

- доступ к файлу и общей папке;[cite:1]
- `file_comment_settings.shared_comments_override`;
- `shared_folder_comment_settings.shared_comments_mode`, если override=`inherit`.

Формула:

- `override=enabled` → shared comments allowed;
- `override=disabled` → shared comments denied;
- `override=inherit` + `folder_mode=enabled` → allowed;
- `override=inherit` + `folder_mode=disabled` → denied;
- `override=inherit` + `folder_mode=inherit_for_items` → allowed для файлов внутри этой папки.

### Общая папка

Для `shared_folder` shared-комментирование разрешено, если `shared_comments_mode != disabled`.

### Локальная папка

Shared-комментарии отсутствуют по определению. Допускаются только private notes.

## API-методы

Ниже приведён REST-вариант API для v2.

### Получение тредов

#### `GET /api/v1/comment-threads`

Query params:

- `targetType=file|shared_folder|local_folder|file_version`
- `targetId=<id>`
- `scope=shared|private|all`
- `contextSharedFolderId=<id|null>`

Возвращает доступные треды и effective policy.

Пример ответа:

```json
{
  "target": {
    "type": "file",
    "id": "file_123"
  },
  "context": {
    "sharedFolderId": "sf_1"
  },
  "policy": {
    "sharedCommentsAllowed": false,
    "source": "shared_folder_mode",
    "sharedCommentsMode": "disabled",
    "fileOverride": "inherit"
  },
  "threads": {
    "shared": {
      "id": "thr_shared_1",
      "commentsCount": 12,
      "unreadCount": 4
    },
    "private": {
      "id": "thr_private_1",
      "commentsCount": 3,
      "unreadCount": 0
    }
  }
}
```

### Создание комментария

#### `POST /api/v1/comments`

```json
{
  "threadId": "thr_shared_1",
  "body": "Проверьте, пожалуйста, финальный вариант",
  "parentCommentId": null,
  "mentions": ["u_2"],
  "contextSharedFolderId": "sf_1"
}
```

Правила:

- для `scope=shared` сервер проверяет effective policy;
- если `target_type=local_folder` и `scope=shared`, возвращается ошибка валидации;
- private-комментарии не зависят от shared policy.

#### Ошибки

```json
{
  "error": "shared_comments_disabled",
  "message": "Общие комментарии для этого объекта отключены"
}
```

### Изменение настроек файла

#### `PATCH /api/v1/files/{fileId}/comment-settings`

```json
{
  "sharedCommentsEnabled": true,
  "sharedCommentsOverride": "inherit",
  "mentionsEnabled": true
}
```

### Изменение настроек общей папки

#### `PATCH /api/v1/shared-folders/{sharedFolderId}/comment-settings`

```json
{
  "sharedCommentsMode": "inherit_for_items",
  "mentionsEnabled": true
}
```

### Изменение настроек локальной папки

#### `PATCH /api/v1/local-folders/{localFolderId}/comment-settings`

```json
{
  "privateCommentsEnabled": true
}
```

### Получение effective policy

#### `GET /api/v1/comment-policy`

Параметры:

- `targetType`
- `targetId`
- `contextSharedFolderId`

Возвращает effective policy, источник правила и доступные действия в UI.

### Прочитанность

#### `POST /api/v1/comment-threads/{threadId}/read`

Отметить тред прочитанным.

#### `GET /api/v1/comment-threads/unread-counters`

Вернуть unread counters для файлов и общих папок батчем, чтобы не делать запрос на каждый объект в `/folders`.[cite:1]

## UX-сценарии

### Сценарий 1. Shared-комментарий к файлу

1. Пользователь открывает карточку файла `/files/:id`.[cite:1]
2. Система показывает shared-thread файла как единый тред объекта файла.
3. Если файл открыт в контексте общей папки, UI также показывает источник policy: правило файла или правило общей папки.
4. Пользователь пишет комментарий, если effective policy разрешает это.

### Сценарий 2. Файл добавлен в «Мои файлы»

1. Участник общей папки открывает файл и выбирает действие **«Добавить в мои файлы»**.[cite:1]
2. Файл появляется в его персональной структуре папок.[cite:1]
3. При открытии комментариев у файла пользователь видит тот же shared-thread файла, что и другие участники.
4. Локальная папка, в которую пользователь поместил файл, не влияет на shared-комментарии файла.

### Сценарий 3. Shared-комментарий к общей папке

1. Пользователь открывает общую папку в `/folders`.[cite:1]
2. В боковой панели **«Обсуждение папки»** отображается shared-thread папки.
3. Если `shared_comments_mode=disabled`, поле ввода shared-комментария заблокировано.
4. Private notes к этой папке остаются доступны пользователю.

### Сценарий 4. Локальная папка

1. Пользователь открывает свою локальную папку в `/folders`.[cite:1]
2. Shared-вкладка для комментариев отсутствует.
3. Доступна только вкладка **«Мои заметки»**.

### Сценарий 5. Переопределение файла в общей папке

1. Общая папка имеет `shared_comments_mode=disabled`.
2. Владелец файла или уполномоченный пользователь меняет `shared_comments_override=enabled`.
3. Для этого файла shared-комментарии становятся доступны, несмотря на запрет на уровне общей папки.
4. UI должен явно показать, что для файла действует собственное правило.

## UX-правила интерфейса

### Карточка файла

В секции `Комментарии`:

- табы `Общие` / `Мои заметки`;
- shared-thread файла;
- бейдж unread;
- composer;
- для владельца файла блок настроек:
  - `Разрешить общие комментарии` — если файл вне общей папки;
  - `Наследовать правило общей папки / Разрешить / Запретить` — если файл открыт в shared-folder context.

### Общая папка

В панели обсуждения общей папки:

- shared-thread папки;
- таб `Мои заметки`;
- селектор политики:
  - `Разрешены`
  - `Запрещены`
  - `Наследуются файлами`

### Локальная папка

В локальной папке:

- только private notes;
- shared controls отсутствуют полностью.

### Тексты состояний

- `Общие комментарии для этой папки отключены`.
- `Комментарии к этому файлу запрещены правилом общей папки`.
- `Для этого файла задано собственное правило комментариев`.
- `Ваши заметки видны только вам`.

## Список экранов для разработки

### 1. Карточка файла `/files/:id`

Основание: карточка файла уже содержит просмотр, описание, теги, доступ, активность и версии.[cite:1] Нужно добавить:

## Нефункциональные требования

### Производительность

- Shared-thread файла должен быть единым и кэшируемым по `file_id`.
- `/folders` должен получать unread counters батчем, а не по одному запросу на объект.[cite:1]
- Effective policy должна вычисляться сервером и отдаваться готовым полем в API, чтобы не дублировать бизнес-логику на клиенте.

### Безопасность

- Нельзя создавать shared-thread для `local_folder`.
- Нельзя получить private-thread другого пользователя.
- Упоминать можно только пользователей, у которых есть доступ к соответствующему файлу или общей папке.[cite:1]
- При отзыве доступа к файлу или исключении из общей папки пользователь теряет доступ и к shared-комментариям соответствующей сущности.[cite:1]

### Аудит

- Любое изменение `shared_comments_enabled`, `shared_comments_override` и `shared_comments_mode` должно логироваться в `comment_audit_log`.
- Создание, редактирование и удаление комментариев также логируется.

## Критерии приёмки

1. Пользователь может создать shared-комментарий к файлу, если effective policy разрешает это.[cite:1]
2. Один и тот же файл имеет единый shared-thread для всех пользователей, независимо от того, в какой локальной папке он у них расположен после `Добавить в мои файлы`.[cite:1]
3. Пользователь может создать shared-комментарий к общей папке, если `shared_comments_mode != disabled`.[cite:1]
4. Пользователь не может создать shared-комментарий к локальной папке.
5. Пользователь всегда может создать private note к файлу, общей папке и локальной папке, если имеет доступ к объекту.
6. Владелец файла может менять `shared_comments_enabled` и `shared_comments_override`.
7. Владелец общей папки или уполномоченный пользователь может менять `shared_comments_mode`.[cite:1]
8. При `shared_comments_mode=disabled` shared-комментарии к общей папке запрещены, а private notes остаются доступны.
9. При `shared_comments_override=enabled` файл может быть комментируемым даже если политика общей папки запрещает shared comments.
10. При `shared_comments_override=disabled` shared-комментарии к файлу запрещены даже если общая папка разрешает обсуждение.

## Phase breakdown

### Phase 1 / MVP

- shared-комментарии к файлам;
- shared-комментарии к общим папкам;
- private-комментарии к файлам, общим папкам и локальным папкам;
- `shared_comments_enabled` для файла;
- `shared_comments_mode` для общей папки;
- `shared_comments_override` для файла;
- unread counters;
- mentions;
- notifications.

### Phase 2

- комментарии к версиям файла;
- вложения в комментариях;
- поиск по комментариям;
- дополнительные permission flags per participant.

### Phase 3

- реакции;
- pinned comments;
- экспорт обсуждения;
- расширение модели при появлении шаринга локальных папок.

## Итог

В версии v2 модель комментариев приводит систему к консистентной архитектуре: shared-комментарии существуют только у действительно общих сущностей — файла и общей папки, а локальная папка остаётся private-only пространством заметок пользователя.[cite:1] Введение `shared_comments_mode` для общей папки в MVP оправдано, потому что текущий DeliFile уже различает локальные и общие папки, позволяет совместно работать с файлами в общей папке и отдельно сохранять такие файлы в персональную структуру пользователя, что требует отделения shared-thread файла от локального пути размещения.[cite:1]
