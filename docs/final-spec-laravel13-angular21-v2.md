.# Итоговая спецификация MVP: backend на Laravel 13 и web-клиент на Angular 21

Документ фиксирует согласованную основу MVP для дальнейшей разработки: стек, продуктовую логику, пользовательские сценарии, экраны, API backend, привязку форм и кнопок, единый формат API response, а также правила разработки в стеке Laravel 13 + Angular 21.[file:1][file:2][file:3]

## 1. Зафиксированный стек

### Основной стек MVP

| Слой | Технология | Решение |
|---|---|---|
| Backend API | Laravel 13 | Основной backend, REST API, бизнес-логика, очереди, scheduler, работа с S3.[file:3] |
| Язык backend | PHP 8.3+ | Базовый runtime для Laravel 13, использовать актуальную стабильную ветку PHP 8.3+.[file:3] |
| Auth для web | Laravel Sanctum | SPA-аутентификация для Angular-клиента, работа через cookie-based session для first-party web-клиента.[file:2][file:3] |
| Web client | Angular 21 | Основной web-интерфейс MVP.[file:3] |
| Angular architecture | Standalone API | Без NgModule для новых экранов и feature-частей, использовать современный standalone-подход.[file:3] |
| UI state | Signals + RxJS | Signals для локального UI state, RxJS для async-flow, HTTP и стримов данных.[file:3] |
| HTTP | Angular HttpClient + interceptors | Авторизация, обработка 401/403, централизованные ошибки, retry только там, где это безопасно.[file:3] |
| Хранилище файлов | Amazon S3 private bucket | Файлы хранятся приватно, скачивание только через backend и временные signed URL.[file:1][file:3] |
| Шифрование MVP | Private S3 + SSE + серверно-управляемое прикладное шифрование | Полноценное strict end-to-end шифрование в MVP не делать.[file:2][file:3] |
| Очереди | Laravel Queue | Фоновые задачи: очистка истекших файлов, логи, уведомления, обработка post-upload действий.[file:3] |
| Планировщик | Laravel Scheduler | TTL-проверки, очистка expired-данных, технические периодические задачи.[file:3] |
| Desktop later | Electron + Angular | Не входит в web MVP, но архитектура должна это учитывать.[file:2][file:3] |
| Mobile later | React Native | Не входит в текущий документ, но backend проектируется общим для web/mobile/desktop.[file:1][file:3] |

### Технологические решения по продукту

- Регистрация и вход: номер телефона + пароль.[file:1][file:2]
- После первой успешной авторизации на доверенном устройстве можно предложить быстрый вход по PIN, но PIN не заменяет основной пароль и не используется как самостоятельный серверный логин.[file:2]
- Файл сначала загружается в аккаунт владельца, и только после завершения загрузки над ним доступны действия шаринга: выдать ссылку, открыть доступ контакту, зафиксировать, добавить в избранное, разложить по папкам и тегам.[file:3]
- Сущность `file` является первичной; `share_link`, `contact_share`, `saved/pinned access` — вторичными действиями над уже загруженным файлом.[file:3]

## 2. Общая суть приложения и user flow

### Суть приложения

Приложение — это личное файловое пространство с быстрым обменом доступом: пользователь загружает файл в свой аккаунт, видит его на других своих устройствах после авторизации, а затем может либо скачать его сам, либо передать доступ контакту, либо создать публичную ссылку.[file:1][file:3]

### Основные сценарии пользователя

1. Пользователь регистрируется по номеру телефона и паролю.[file:1][file:2]
2. Пользователь входит в систему и попадает на главный экран со списком своих файлов и зоной загрузки.[file:3]
3. Пользователь выбирает файл и нажимает кнопку загрузки.[file:3]
4. После успешной загрузки файл появляется в аккаунте владельца и доступен ему на всех его устройствах после авторизации.[file:3]
5. Уже после загрузки пользователь может открыть карточку файла и выполнить одно из действий: скачать, передать контакту, создать ссылку, зафиксировать, добавить в избранное, перенести в папку, назначить теги.[file:3]
6. Контакт, которому открыт доступ, получает доступ либо как зарегистрированный пользователь, либо через сценарий приглашения/дальнейшей регистрации.[file:1][file:3]
7. Получатель может логически сохранить файл у себя навсегда без физического дублирования объекта в S3, а владелец должен видеть факт такого сохранения в истории действий.[file:3]

### Главный user flow MVP

#### Flow A. Регистрация и вход

1. Открытие web-приложения.
2. Переход на экран входа.
3. Переход на регистрацию при отсутствии аккаунта.
4. Ввод номера телефона и пароля.
5. Создание аккаунта.
6. Опциональный экран предложения настроить быстрый вход по PIN для доверенного устройства, как вторичную функцию после полноценной авторизации.[file:2]
7. Переход на главный экран приложения.[file:2][file:3]

#### Flow B. Загрузка файла

1. Пользователь находится на главном экране.
2. Нажимает «Выбрать файл» или использует drag-and-drop.[file:3]
3. Приложение показывает выбранный файл и кнопку «Загрузить».[file:3]
4. После нажатия «Загрузить» запускается upload flow: инициализация загрузки, прямая загрузка в S3, подтверждение завершения.[file:3]
5. После успеха пользователь попадает в карточку файла или остается на списке со snackbar/уведомлением и быстрыми действиями.[file:3]

#### Flow C. Доступ к своему файлу с другого устройства

1. Пользователь входит на другом устройстве в тот же аккаунт.[file:3]
2. Открывает список файлов.
3. Видит загруженный ранее файл.
4. Открывает карточку.
5. Нажимает «Скачать».
6. Backend проверяет права и выдает безопасную временную ссылку на скачивание.[file:3]

#### Flow D. Передача файла контакту

1. Пользователь открывает карточку файла.[file:3]
2. Нажимает «Передать контакту».
3. В открывшемся экране/модальном окне ищет контакт по имени или телефону.[file:3]
4. Выбирает контакт.
5. Нажимает «Открыть доступ».
6. Backend создает запись доступа к файлу для целевого пользователя/контакта.[file:3]
7. При необходимости запускается сценарий приглашения незарегистрированного контакта.[file:1][file:3]

#### Flow E. Создание публичной ссылки

1. Пользователь открывает карточку файла.[file:3]
2. Нажимает «Создать ссылку».
3. Указывает параметры ссылки, в MVP минимум TTL, по умолчанию 12 часов.[file:1][file:3]
4. Нажимает «Создать».
5. Система создает запись ссылки, возвращает URL и дает действия «Скопировать», «Отключить».[file:3]

#### Flow F. Логическая фиксация файла

1. Владелец или получатель, имеющий доступ к файлу, открывает карточку файла.[file:3]
2. Нажимает «Зафиксировать» / «Сохранить у себя».
3. Система не создает вторую физическую копию файла, а добавляет логическую постоянную привязку доступа к одному и тому же объекту в S3.[file:3]
4. Владельцу фиксируется событие, что другой пользователь сохранил файл у себя навсегда.[file:3]

## 3. Экраны пользователя: поля и кнопки

Ниже описаны именно web-экраны MVP для Angular 21.

### 3.1. Экран входа

**Назначение:** вход по номеру телефона и паролю.[file:1][file:2]

**Поля:**
- Номер телефона.
- Пароль.
- Checkbox «Запомнить устройство» — опционально.

**Кнопки и действия:**
- «Войти».
- «Зарегистрироваться».
- «Забыли пароль?»
- «Войти по PIN» — только если доверенное устройство и функция уже настроена; для web оставить как отложенную/опциональную возможность, не обязательную для MVP.[file:2]

### 3.2. Экран регистрации

**Назначение:** создание аккаунта.[file:1][file:2]

**Поля:**
- Номер телефона.
- Пароль.
- Подтверждение пароля.
- Согласие с правилами/политикой — если требуется юридически.

**Кнопки и действия:**
- «Создать аккаунт».
- «У меня уже есть аккаунт».

### 3.3. Экран предложения PIN после регистрации/первого входа

**Назначение:** предложение включить быстрый вход на доверенном устройстве как вторичную локальную функцию.[file:2]

**Поля:**
- PIN.
- Повтор PIN.

**Кнопки и действия:**
- «Включить быстрый вход по PIN».
- «Пропустить сейчас».

**Примечание:** для web данный экран может быть отключаемым feature flag; в MVP достаточно зафиксировать поддержку сценария в общей архитектуре, но не делать его обязательным в web первой версии.[file:2]

### 3.4. Главный экран / Мои файлы

**Назначение:** точка входа после логина, список файлов и быстрый доступ к загрузке.[file:3]

**Блоки:**
- Header приложения.
- Кнопка «Выбрать файл».
- Drag-and-drop зона для загрузки.[file:3]
- Список последних/всех файлов.[file:3]
- Фильтры и поиск — если включаются в MVP.
- Переключатели разделов: «Мои», «Полученные», «Избранное», при необходимости.[file:3]

**Поля / элементы фильтрации:**
- Поиск по имени файла.
- Фильтр по статусу: активные, истекающие, закрепленные.
- Фильтр по папке/тегу — если включено в MVP.

**Кнопки и действия:**
- «Выбрать файл».
- «Загрузить» — появляется после выбора файла.[file:3]
- На карточке файла: «Открыть», «Скачать», «Передать контакту», «Создать ссылку», «Зафиксировать», «В избранное».[file:3]
- Меню профиля: «Настройки», «Сессии/устройства», «Выйти».

### 3.5. Экран загрузки файла

**Назначение:** короткий и понятный upload flow без перегрузки полями.[file:3]

**Поля / элементы:**
- Название выбранного файла.
- Размер.
- MIME/type — readonly.
- Progress bar.[file:3]

**Кнопки и действия:**
- «Загрузить».
- «Отменить».
- После успеха: «Перейти к файлу», «Скопировать ссылку» — только если ссылка уже создана отдельно, по умолчанию не создавать автоматически.[file:3]

**Назначение:** главный рабочий экран для действий над файлом.[file:3]

- Кто имеет доступ.[file:3]

## 4. Endpoint'ы backend и их функциональность

Ниже — рекомендуемая структура REST API для Laravel 13. Все защищенные endpoint'ы работают под авторизацией Sanctum, кроме публичных link-flow и endpoint'ов регистрации/восстановления доступа.[file:2][file:3]

### 4.1. Единый формат API response

Все API response в проекте должны быть приведены к единому виду.[file:1][file:2][file:3]

#### Базовый формат успешного ответа

```json
{
  "result": "success",
  "message": "Result message",
  "data": {
    "some_key": "some_value"
  }
}
```

#### Базовый формат ответа с ошибкой

```json
{
  "result": "error",
  "message": "Error message",
  "data": {
    "errors": {
      "field": [
        "Validation message"
      ]
    },
    "code": "VALIDATION_ERROR"
  }
}
```

#### Обязательные правила response contract

- Поле `result` обязательно во всех JSON-ответах API.
- Для успешных операций использовать `result: "success"`.
- Для ошибок использовать `result: "error"`.
- Поле `message` обязательно во всех JSON-ответах API и должно содержать короткое человекочитаемое описание результата операции.
- Поле `data` обязательно во всех JSON-ответах API, даже если полезная нагрузка минимальна.
- Если данных нет, использовать пустой объект `data: {}`.
- Нельзя возвращать на одних endpoint'ах «сырые» модели, а на других кастомную структуру; формат должен быть единым для всего проекта.
- Для списков внутри `data` использовать именованные ключи, например `items`, `pagination`, `file`, `user`, `link`, `sessions`.
- Для ошибок валидации Laravel приводить стандартный validation response к общему контракту проекта.
- Angular-клиент должен ожидать только этот унифицированный response contract.

#### Примеры типовых `data`

- Один объект: `data: { "file": { ... } }`
- Список: `data: { "items": [ ... ], "pagination": { ... } }`
- Успешное действие без сложной нагрузки: `data: { "id": "uuid" }`
- Пустой успешный ответ: `data: {}`

### 4.2. Auth

| Method | Endpoint | Назначение |
|---|---|---|
| POST | `/api/auth/register` | Регистрация пользователя по номеру телефона и паролю.[file:1][file:2][file:3] |
| POST | `/api/auth/login` | Вход по номеру телефона и паролю.[file:1][file:2][file:3] |
| POST | `/api/auth/logout` | Выход из текущей сессии.[file:3] |
| POST | `/api/auth/logout-all` | Завершение всех активных сессий пользователя.[file:2][file:3] |
| GET | `/api/auth/me` | Получение текущего пользователя, ролей, базовых настроек.[file:3] |
| GET | `/api/auth/sessions` | Список активных сессий/устройств.[file:2][file:3] |
| DELETE | `/api/auth/sessions/{sessionId}` | Завершение конкретной сессии/устройства.[file:3] |
| POST | `/api/auth/password/forgot` | Старт восстановления пароля, в будущем через SMS flow.[file:2][file:3] |

| POST | `/api/auth/password/change` | Смена пароля авторизованным пользователем.[file:2][file:3] |

### 4.3. Files

#### Базовые endpoints

| Method | Endpoint | Назначение |
|---|---|---|
| GET | `/api/files` | Получение списка файлов пользователя: собственные, полученные, сохраненные; с фильтрами и пагинацией.[file:3] |
| GET | `/api/files/{fileId}` | Получение детальной карточки файла.[file:3] |
| DELETE | `/api/files/{fileId}` | Логическое удаление файла владельцем с проверкой зависимостей доступа.[file:3] |

#### Upload flow

| Method | Endpoint | Назначение |
|---|---|---|
| POST | `/api/files/init-upload` | Инициализация загрузки: имя, размер, mime, checksum; создание записи файла в статусе `uploading`, генерация S3 presigned upload URL.[file:3] |
| POST | `/api/files/complete-upload` | Подтверждение завершения прямой загрузки в S3, перевод файла в `available`, создание owner access, запись лога.[file:3] |
| POST | `/api/files/{fileId}/cancel-upload` | Отмена незавершенной загрузки и очистка черновых данных. |

#### Работа с файлом

| Method | Endpoint | Назначение |
|---|---|---|
| POST | `/api/files/{fileId}/download` | Проверка права доступа и выдача signed download URL.[file:3] |
| POST | `/api/files/{fileId}/pin` | Закрепление файла владельцем или логическое сохранение у получателя.[file:3] |

## 6. Инструкция и правила разработки именно в Laravel 13 + Angular 21

Ниже — обязательные правила для команды разработки. Они должны использоваться как постоянный стандарт проекта.

### 6.1. Архитектурные принципы проекта

- Backend строится как API-first приложение на Laravel 13.[file:3]
- Web-клиент строится как standalone SPA на Angular 21 без legacy-подхода через NgModule для новых feature-модулей.[file:3]
- Весь MVP проектируется вокруг первичной сущности `file`, а не вокруг передачи как отдельного главного процесса.[file:3]
- Upload flow должен быть трехшаговым: `init-upload` → direct upload в S3 → `complete-upload`.[file:3]
- Доступ к файлам всегда проверяется backend'ом; S3 остается private, скачивание выполняется только через подписанные URL после проверки прав.[file:1][file:3]
- Фиксация файла у другого пользователя реализуется логически через модель доступа, без физического дублирования файла в S3.[file:3]
- Все API response должны использовать единый response contract проекта.[file:1][file:2][file:3]

### 6.2. Правила backend на Laravel 13

#### Структура слоев

- `Controllers` — только прием запроса, валидация через Form Request, вызов application/service слоя, возврат API Resource.
- `FormRequest` — вся входная валидация.
- `Services` / `Actions` — бизнес-логика use case уровня: upload init, upload complete, share to contact, create link, pin/unpin, download permission check.
- `Policies` — централизованная проверка доступа к файлам, ссылкам, контактам и действиям.[file:3]
- `Jobs` — фоновые операции и отложенные действия.
- `Resources` — единый формат JSON-ответов.
- `Enums` — статусы файла, типы доступа, типы событий активности.

#### Рекомендуемые Laravel-практики

- Использовать `Laravel Sanctum` для SPA auth в web-клиенте.[file:2][file:3]
- Для публичных endpoint'ов link-flow четко отделять guest routes от auth routes.[file:3]
- Использовать `FormRequest` вместо ручной валидации в controller.
- Использовать `API Resources` или единый response builder вместо возврата сырых моделей.
- Использовать `Policies` и `Gates` вместо точечных проверок в контроллерах.
- Очереди для тяжелых и фоновых операций: очистка истекших файлов, уведомления, аудит, постобработка.[file:3]
- `Scheduler` для TTL cleanup и регулярной обработки lifecycle файлов.[file:3]
- Логически важные операции оборачивать в `DB::transaction()`.
- Для файлов и прав доступа использовать UUID/ULID как внешние идентификаторы.
- Версионировать API через `/api/v1/...`, даже если в MVP версия одна.
- Сделать единый helper/trait/response service для генерации API-ответов строго по контракту `{result, message, data}`.
- Стандартизовать также error response, validation response и not found response через общий слой обработки исключений.

#### Модели и сущности, которые зафиксировать сразу

Минимальный набор доменных сущностей:
- `User`.[file:3]
- `DeviceSession` / `Device`.[file:2][file:3]
- `File`.[file:3]
- `FileUserAccess`.[file:3]
- `ShareLink`.[file:3]
- `Contact`.[file:3]
- `ContactShare`.[file:3]
- `Folder`.[file:3]
- `Tag`.[file:3]
- `Favorite`.[file:3]
- `ActivityLog`.[file:3]

#### Статусы и типы, которые оформить через enum

- `FileStatus`: `uploading`, `available`, `processing`, `expired`, `deleted`.
- `AccessType`: `owner`, `shared`, `saved`.[file:3]
- `ShareLinkStatus`: `active`, `disabled`, `expired`.
- `ActivityType`: `uploaded`, `downloaded`, `shared_to_contact`, `share_revoked`, `link_created`, `link_disabled`, `pinned`, `unpinned`, `favorited`, `moved_to_folder`, `tag_updated`, `saved_by_recipient`.[file:3]

#### Безопасность backend

- Не использовать публичные S3 URL для файлов.[file:1][file:3]
- Все скачивания — через backend permission check и только потом signed URL.[file:3]
- В MVP не реализовывать strict end-to-end шифрование; использовать private S3, SSE и при необходимости серверно-управляемое прикладное шифрование.[file:2][file:3]
- PIN не должен быть самостоятельным backend-логином.[file:2]
- PIN — только вторичный механизм доступа на доверенном устройстве.[file:2]
- Для чувствительных операций при необходимости требовать повторный ввод пароля.[file:2]

### 6.3. Правила frontend на Angular 21

#### Базовый Angular-подход

- Использовать standalone components по умолчанию.[file:3]
- Не создавать новые NgModule без очень веской причины.
- Использовать `provideRouter`, `provideHttpClient`, `withInterceptors`, `withComponentInputBinding` и другие современные provider-based API Angular 21.
- Для локального состояния экрана использовать `signals`.
- Для async-операций, HTTP, стримов, search, polling — использовать `RxJS`.
- Реактивные формы строить через `ReactiveFormsModule`/standalone imports, а не template-driven forms для ключевых бизнес-форм.
- Строгая типизация DTO, response model, form model и UI model.
- На frontend зафиксировать единый тип API-ответа, например `ApiResponse<T> = { result: 'success' | 'error'; message: string; data: T }`.

#### Рекомендуемая структура Angular-приложения

```text
src/app/
  core/
    api/
    auth/
    guards/
    interceptors/
    layout/
    services/
    tokens/
  shared/
    ui/
    pipes/
    directives/
    utils/
    models/
  features/
    auth/
    files/
    contacts/
    activity/
    settings/
```

#### Правила по feature-архитектуре

- Каждый feature хранит свои pages, components, dialogs, facades, api-clients, models и routes локально.
- Переиспользуемые UI-компоненты выносить в `shared/ui`.
- Бизнес-специфичные компоненты не поднимать в shared преждевременно.
- Для экранов использовать container/presenter подход там, где это уменьшает связанность.
- Dialog/overlay сценарии для «Передать контакту» и «Создать ссылку» проектировать как самостоятельные standalone feature-dialog components.

#### Работа с HTTP и auth

- Все HTTP-вызовы централизовать в typed API services.
- Использовать interceptor для обработки auth-state, 401, 403, общих ошибок.
- Не смешивать прямой `HttpClient` внутри компонентов, кроме очень простых локальных случаев; основной путь — facade/service.
- Для upload flow выделить отдельный `FileUploadService`, который умеет:
  - вызвать `init-upload`;
  - выполнить upload в S3;
  - вызвать `complete-upload`;
  - отдавать progress и error state в UI.[file:3]
- Весь parsing ответов backend строить только вокруг единого `ApiResponse<T>`.

#### Работа с формами

- Все бизнес-формы — reactive forms.
- Вся форма имеет typed form model.
- Ошибки валидации показывать у поля и на submit.
- Кнопки submit блокировать во время pending state.
- Не допускать двойного submit.
- Статусы загрузки и ошибок делать явными, не скрытыми.

#### Роутинг Angular 21

Минимальная маршрутная карта:
- `/login`
- `/register`
- `/pin-setup` — опционально/feature-flag.[file:2]
- `/files`
- `/files/:id`
- `/contacts`
- `/activity`
- `/settings/security`

#### UI и UX правила

- Upload flow должен быть коротким: выбрать файл → загрузить → перейти к карточке файла.[file:3]
- Не перегружать форму загрузки папками, тегами, адресатами до завершения самой загрузки.[file:3]
- Основные действия по файлу размещать в карточке файла и в quick actions списка файлов.[file:3]
- Действия, меняющие доступ к файлу, подтверждать и логировать.[file:3]
- Все destructive actions сопровождать confirm dialog.
- Для long-running процессов показывать progress bar, pending state и понятные ошибки.

### 6.4. Правила интеграции backend и frontend

- Все контракты описывать через OpenAPI/Swagger либо отдельный API contract markdown рядом с backend.
- Имена DTO должны совпадать по смыслу между backend и frontend.
- Единый формат ошибок API фиксировать сразу.
- Для всех успешных и ошибочных ответов использовать общий контракт `{result, message, data}`.
- Для списков использовать единый формат пагинации внутри `data.pagination`.
- Для дат использовать ISO 8601.
- Для file id, contact id, link id использовать строковые UUID/ULID.
- Для статусов и enum значений использовать одинаковые константы по обе стороны.
- Любые новые endpoint'ы считаются корректными только если они сразу соответствуют общему response contract и naming contract.

### 6.5. Что не делать в проекте

- Не делать strict end-to-end шифрование в MVP.[file:2][file:3]
- Не делать обязательный PIN-login для web MVP.[file:2]
- Не превращать upload в перегруженный мастер с передачей, тегами и папками до завершения самой загрузки.[file:3]
- Не делать публичные S3 файлы.[file:1][file:3]
- Не дублировать физически файл при логическом сохранении получателем.[file:3]
- Не писать бизнес-логику в Angular-компонентах и Laravel-контроллерах.
- Не использовать устаревший Angular-подход как основу нового кода, если есть современный standalone API.
- Не допускать разнобой форматов API response между endpoint'ами.

## 7. Зафиксированные продуктовые правила MVP

- Основной вход: номер телефона + пароль.[file:1][file:2]
- Восстановление PIN: через пароль.[file:2]
- Восстановление пароля: через SMS flow, даже если в первой реализации это будет stub/mock-провайдер.[file:2]
- PIN является вторичным локальным способом быстрого входа только после основной авторизации.[file:2]
- Файл по умолчанию имеет TTL 12 часов, если не зафиксирован и если не действует иная логика сохранения.[file:1][file:3]
- Владелец файла всегда имеет полный доступ.[file:3]
- Получатель по контакту получает доступ после назначения или после завершения сценария регистрации/приглашения.[file:3]
- Получатель может логически сохранить файл у себя навсегда без физической копии.[file:3]
- Владелец должен видеть факт, что другой пользователь сохранил его файл у себя навсегда.[file:3]
- Физическое удаление из S3 возможно только когда истек TTL, нет активных ссылок и нет ни одного актуального логического сохранения/доступа, которое продлевает жизненный цикл файла.[file:3]

## 10. Ключевые инструкции для всей разработки

Этот блок считать обязательным для постоянного использования на всем этапе проекта.[file:1][file:2][file:3]

### 10.1. Что за приложение мы делаем

Разрабатывается web MVP личного файлового пространства с быстрым обменом доступом, где пользователь сначала загружает файл в свой аккаунт, затем может скачать его с другого своего устройства, передать доступ контакту или создать публичную ссылку.[file:1][file:3]

### 10.2. Зафиксированный стек, который не менять без отдельного решения

- Backend: Laravel 13 + PHP 8.3+.[file:3]
- Auth: Laravel Sanctum.[file:2][file:3]
- Web: Angular 21 standalone + signals + RxJS + reactive forms.[file:3]
- Storage: private S3 + signed URL.[file:1][file:3]
- Background processing: Laravel Queue + Scheduler.[file:3]
- Шифрование MVP: private S3 + SSE + серверно-управляемое прикладное шифрование, без strict end-to-end.[file:2][file:3]

### 10.3. Главная продуктовая логика, которую нельзя ломать

- Сначала загрузка файла, потом все остальные действия.[file:3]
- Первичная сущность — `file`, а не `transfer`.[file:3]
- После загрузки файл доступен владельцу на его других устройствах после авторизации.[file:3]
- Share-to-contact и public link создаются только после успешной загрузки файла.[file:3]
- Получатель может сохранить файл у себя логически, без физической копии объекта.[file:3]
- Владелец должен видеть факт такого сохранения в истории.[file:3]

### 10.4. Единый API response contract

Все backend endpoint'ы без исключения обязаны возвращать JSON в формате:

```json
{
  "result": "success",
  "message": "Result message",
  "data": {
    "some_key": "some_value"
  }
}
```

Для ошибок:

```json
{
  "result": "error",
  "message": "Error message",
  "data": {
    "code": "ERROR_CODE",
    "errors": {}
  }
}
```

Дополнительные правила:
- `data` обязательно всегда.
- Пустой ответ тоже должен быть с `data: {}`.
- Angular работает только с этим контрактом.
- Любой новый endpoint без этого контракта считается нарушением стандарта проекта.

### 10.5. Обязательные архитектурные правила

- В Laravel controller не содержит бизнес-логику.
- В Angular component не содержит тяжелую бизнес-логику и прямую хаотичную работу с API.
- Для новых Angular частей использовать только standalone-подход, если нет очень веской причины иначе.[file:3]
- Для форм использовать reactive forms.
- Для API и моделей использовать строгую типизацию.
- Для доступа к файлам всегда проверять права на backend до выдачи signed URL.[file:3]
- Создаваемые переменные при разработке должны быть информативные и иметь не менее 3 знаков.

- Не внедрять strict end-to-end encryption в MVP.[file:2][file:3]
