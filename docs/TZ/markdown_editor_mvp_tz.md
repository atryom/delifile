# Техническое задание: MVP поддержки создания и редактирования Markdown-документов

## 1. Цель

Реализовать в проекте поддержку создания, открытия, редактирования и сохранения текстовых файлов формата `.md` через встроенный визуальный редактор.

Файлы `.md` должны храниться в существующем S3-хранилище на общих основаниях с остальными файлами проекта. Для файлов типа Markdown необходимо добавить отдельный прикладной сценарий обработки: вместо режима только просмотра и скачивания пользователь должен иметь возможность открыть документ в редакторе, изменить его содержимое и сохранить обновлённую версию обратно в S3.

## 2. Границы MVP

В рамках MVP реализуется только один режим работы с документом: визуальный редактор без split-view, без отображения исходного Markdown и без отдельной панели preview.

В качестве редактора используется Tiptap на базе шаблона Simple Editor.

Markdown является каноническим форматом хранения документа.

Поддержка BlockNote, совместного редактирования, merge-конфликтов, истории изменений внутри редактора, комментариев и расширенной блочной модели в MVP не входит.

## 3. Архитектурные решения

### 3.1 Версионирование API

Все новые маршруты должны использовать текущую стилистику API проекта с префиксом `/api/v1`.

### 3.2 Отдельный ресурс Documents

Для работы с markdown-документами необходимо использовать отдельный `DocumentController`, а не расширять существующий `FileController` логикой редактирования содержимого.

Причина: сценарий редактируемого документа является отдельной прикладной сущностью и в будущем, вероятно, будет расширяться дополнительной функциональностью.

### 3.3 Канонический формат

Каноническим форматом хранения документа является Markdown.

В S3 не должно храниться отдельное HTML-представление документа как основной источник данных.

### 3.4 Stable URL для изображений

S3-хранилище проекта является приватным.

Внутри markdown-документа запрещено сохранять временные presigned URL, так как они имеют ограниченный срок действия и не подходят для постоянных ссылок внутри документа.

Для изображений, вставляемых в документ, необходимо использовать стабильные application URL, например:

`/api/v1/files/{id}/content`

В markdown-документе сохраняется именно такой стабильный URL.

При обращении к нему backend обязан:
- проверить права доступа пользователя к файлу;
- после проверки либо проксировать содержимое файла, либо выполнить контролируемую выдачу актуального содержимого через внутренний механизм доступа к S3.

Публичный бакет или хранение presigned URL внутри документа в рамках MVP не используются.

### 3.5 Защита от параллельного редактирования

Для снижения риска конфликтов необходимо использовать комбинированный подход:
- pessimistic lock на время редактирования документа;
- проверку версии документа при сохранении по актуальному `etag` объекта в S3.

Это означает:
- при входе пользователя в режим редактирования создаётся lock документа;
- пока lock активен, другой пользователь не должен получать обычный режим редактирования;
- пользователь, открывающий уже заблокированный документ, должен получать сообщение о том, что файл сейчас редактируется другим пользователем;
- только владелец документа (`access_type = owner`) имеет право принудительно забрать lock у другого пользователя;
- даже при наличии lock при сохранении всё равно должна происходить проверка `etag`, чтобы избежать записи поверх более новой версии.

### 3.6 Жизненный цикл lock

Для предотвращения зависания lock необходимо реализовать следующий жизненный цикл:
- lock создаётся при входе в режим редактирования;
- TTL lock составляет **5 минут**;
- клиент отправляет heartbeat каждые **60 секунд** для продления TTL;
- при штатном выходе из редактора lock освобождается явным запросом;
- при истечении TTL без heartbeat lock считается протухшим и может быть получен другим пользователем.

При потере lock (TTL истёк или lock забрали через takeover) heartbeat возвращает `423 Locked` с машиночитаемым полем `reason`:
- `LOCK_EXPIRED` — TTL истёк без heartbeat; frontend предлагает пользователю заново получить lock;
- `LOCK_TAKEN_OVER` — lock принудительно забран владельцем; frontend переводит документ в read-only и показывает сообщение с именем нового редактора.

### 3.7 Модель прав на редактирование документа

Право редактировать markdown-документ определяется через поле `can_edit` в таблице `file_user_access`:
- пользователь с `access_type = owner` всегда может редактировать документ;
- пользователь с `access_type = shared` может редактировать только при `can_edit = true`; по умолчанию при выдаче доступа `can_edit = false`;
- пользователь с `access_type = saved` не может редактировать документ.

Изменение значения `can_edit` для уже выданного shared-доступа выполняется через отдельный endpoint управления правами, не через endpoint шаринга.

Пользователи без права редактирования (`canEdit = false`) могут открывать markdown-документ в режиме чтения через тот же экран редактора. В этом режиме редактирование, toolbar-действия, получение lock и сохранение недоступны. Файл не должен сводиться к скачиваемому объекту.

## 4. Основной пользовательский сценарий

### 4.1 Создание документа

Пользователь должен иметь возможность создать новый текстовый файл типа `.md` из файлового менеджера проекта.

Кнопка создания должна называться **«Создать заметку»** и располагаться рядом с действием **«Загрузить файл»**.

При создании:
- создаётся новый файловый объект в общей файловой модели;
- задаётся имя файла с расширением `.md`;
- создаётся запись метаданных файла;
- создаётся пустой или начально заполненный объект в S3;
- файл помечается как доступный для редактирования через встроенный редактор.

Привязка к `parent_id` в MVP не требуется.

Файл создаётся как самостоятельный объект файловой системы и далее может быть перемещён существующими механизмами проекта.

### 4.2 Открытие документа

При открытии файла типа `.md`:
- система не должна переводить пользователя в стандартный режим скачивания файла;
- должен открываться экран редактора документа;
- содержимое файла должно загружаться из S3 как текст Markdown;
- редактор должен отображать документ в визуальном виде.

Если документ уже находится на редактировании у другого пользователя и текущий пользователь не выполнил принудительный захват, обычный режим редактирования не должен открываться.

### 4.3 Сохранение документа

При сохранении:
- текущее содержимое редактора сериализуется в Markdown;
- обновлённое содержимое отправляется на backend;
- backend сохраняет содержимое в тот же объект S3;
- после успешного сохранения клиент получает новую версию `etag` и обновлённые метаданные документа.

Если версия документа устарела, клиент должен получить контролируемую ошибку конфликта.

Для MVP достаточно показать пользователю понятное сообщение о конфликте сохранения без реализации автоматического merge.

## 5. Требования к UX

### 5.1 Общий принцип

Редактор должен быть ориентирован на нетехнического пользователя.

Интерфейс должен восприниматься как обычный редактор документа, а не как инструмент для работы с исходным кодом или разметкой.

### 5.2 Обязательные элементы экрана редактора

На экране редактирования должны присутствовать:
- имя файла;
- основная область редактора;
- toolbar с действиями форматирования;
- кнопка сохранения;
- статус сохранения, например: «Сохранено», «Есть несохранённые изменения», «Сохраняем», «Ошибка сохранения»;
- действие вставки изображения из доступных пользователю файлов.

При активном lock другого пользователя должен отображаться информирующий экран или блокирующее сообщение с указанием, кто редактирует документ в данный момент.

Если текущий пользователь является владельцем документа, в этом состоянии должна отображаться кнопка принудительного захвата lock.

Если lock был потерян во время редактирования (истёк TTL или отобран через takeover), экран редактора должен реагировать согласно причине потери:
- при `LOCK_EXPIRED` показать предложение заново получить lock, чтобы продолжить редактирование;
- при `LOCK_TAKEN_OVER` перевести документ в режим чтения и показать сообщение о том, кто забрал документ.

Пользователи без права редактирования (`canEdit = false`) открывают документ в режиме чтения: toolbar, сохранение и lock-действия скрыты или заблокированы.

### 5.3 Поведение редактора

Редактор должен поддерживать:
- загрузку markdown-содержимого при открытии документа;
- работу как визуальный rich-text редактор;
- сохранение содержимого в формате Markdown;
- базовое автосохранение с debounce либо явное сохранение по кнопке, решение определяется реализацией, но в интерфейсе всегда должен быть понятен текущий статус сохранения.

## 6. Поддерживаемые действия toolbar

В MVP необходимо поддержать следующие действия форматирования:
- paragraph;
- heading;
- bold;
- italic;
- strike;
- bullet list;
- ordered list;
- task list;
- blockquote;
- inline code;
- code block;
- link;
- image;
- undo;
- redo.

Допускается отключить функции, которые нестабильно сериализуются в Markdown или не нужны для первого этапа.

Не требуется реализовывать в MVP:
- таблицы;
- выравнивание текста;
- произвольные цвета текста;
- вложенные сложные блоки;
- drag-and-drop layout-блоки;
- комментарии и suggest mode.

## 7. Работа с изображениями

### 7.1 Общий принцип

В редакторе должна быть возможность вставки изображений, уже доступных пользователю по правам в рамках существующей файловой системы.

Изображения хранятся в том же S3, что и остальные файлы проекта.

### 7.2 Область доступных изображений

В список доступных для вставки изображений должны попадать все изображения, которые доступны текущему пользователю в системе по действующим правилам доступа, включая:
- собственные изображения пользователя;
- изображения из расшаренных ресурсов;
- изображения из других доступных пользователю контекстов файловой системы.

Для удобства работы допускается реализовать фильтрацию, поиск и пагинацию.

### 7.3 Сценарий вставки изображения

При выборе действия вставки изображения:
- открывается модальное окно или боковая панель выбора изображений;
- пользователь видит только изображения, доступные ему по правам;
- пользователь может выбрать одно из доступных изображений;
- выбранное изображение вставляется в документ как элемент редактора;
- при сериализации документа ссылка на изображение должна сохраняться в Markdown как стабильный application URL.

### 7.4 Источник изображений

Для вставки должны использоваться только стабильные URL, пригодные для последующего отображения в сохранённом документе.

Необходимо использовать постоянный application URL через backend, а не временный presigned URL.

### 7.5 Получение размеров изображения

Для новых изображений необходимо извлекать ширину и высоту на этапе загрузки и сохранять их в метаданных файла.

Для уже существующих изображений, у которых эти данные отсутствуют, допускается возвращать `null` до момента фонового обогащения данных или повторной обработки.

### 7.6 Дополнительный сценарий загрузки

Если пользователь загружает новое изображение из редактора, редактор должен использовать существующий flow загрузки файлов проекта:
- init-upload;
- прямая загрузка в S3;
- complete-upload.

Дополнительный одношаговый backend-upload endpoint для MVP не требуется.

Редактор может использовать UI-обёртку над существующим upload flow.

## 8. Правила хранения

### 8.1 Хранение в S3

Для каждого markdown-документа:
- создаётся объект в S3;
- объект хранится как текстовый файл с расширением `.md`;
- для объекта должен указываться корректный content type, например `text/markdown` либо эквивалентный текстовый MIME-тип, принятый в проекте.

### 8.2 Метаданные файла

В таблицу `files` необходимо добавить через миграцию следующие поля:
- `is_editable` (bool, default false) — признак редактируемого документа;
- `editor_type` (string, nullable) — тип редактора, для markdown: `markdown`;
- `width` (int, nullable) — ширина изображения в пикселях;
- `height` (int, nullable) — высота изображения в пикселях;
- `etag` (string, nullable) — актуальный ETag объекта в S3, обновляется после каждого сохранения;
- `updated_by` (FK→users, nullable) — пользователь, последним сохранивший документ.

В таблицу `file_user_access` необходимо добавить:
- `can_edit` (bool, default false) — право редактировать документ для пользователей с `access_type = shared`.

Также используются существующие поля модели `files`:
- `id`, `storage_key`, `original_name`, `mime_type`, `size`, `owner_id`, `updated_at`, `content_kind`.

Для markdown-файлов рекомендуется задавать:
- `isEditable = true`;
- `editorType = markdown` или эквивалентное значение, принятое в проекте.

## 9. API-контракт MVP

## 9.1 Создание документа

### Endpoint

`POST /api/v1/documents`

### Назначение

Создание нового markdown-документа.

### Пример запроса

```json
{
  "fileName": "Новый документ.md"
}
```

### Пример ответа

```json
{
  "id": "file_789",
  "fileName": "Новый документ.md",
  "mimeType": "text/markdown",
  "isEditable": true,
  "editorType": "markdown",
  "storageKey": "users/u1/docs/file_789.md",
  "etag": null,
  "content": "",
  "updatedAt": "2026-05-16T07:00:00Z"
}
```

## 9.2 Получение документа для редактирования

### Endpoint

`GET /api/v1/documents/:id`

### Назначение

Получение markdown-содержимого и метаданных документа для открытия в редакторе.

### Пример ответа

```json
{
  "id": "file_789",
  "fileName": "Новый документ.md",
  "mimeType": "text/markdown",
  "isEditable": true,
  "editorType": "markdown",
  "content": "# Заголовок\n\nТекст документа",
  "etag": "\"9b2cf535f27731c974343645a3985328\"",
  "updatedAt": "2026-05-16T07:05:00Z",
  "updatedBy": {
    "id": "user_12",
    "name": "Ivan"
  },
  "lock": {
    "isLocked": true,
    "lockedBy": {
      "id": "user_17",
      "name": "Olga"
    },
    "expiresAt": "2026-05-16T07:20:00Z",
    "canTakeOver": true
  },
  "capabilities": {
    "canEdit": true,
    "canRename": true,
    "canDelete": true,
    "canInsertImages": true,
    "canTakeOverLock": true
  }
}
```

## 9.3 Захват документа на редактирование

### Endpoint

`POST /api/v1/documents/:id/lock`

### Назначение

Создание lock для входа в режим редактирования.

### HTTP-статусы

- `201 Created` — lock успешно создан для текущего пользователя.
- `423 Locked` — документ уже захвачен другим пользователем; тело ответа содержит состояние lock и признак возможности принудительного захвата.

### Сценарий

Если документ свободен, lock создаётся для текущего пользователя, TTL — 5 минут.

Если документ уже захвачен другим пользователем, backend возвращает `423` с текущим состоянием lock. Поле `canTakeOver` будет `true` только для владельца документа.

## 9.4 Продление lock

### Endpoint

`POST /api/v1/documents/:id/lock/heartbeat`

### Назначение

Продление времени жизни активного lock документа. Должен вызываться каждые 60 секунд.

### HTTP-статусы

- `200 OK` — lock успешно продлён.
- `423 Locked` — lock потерян; тело ответа содержит машиночитаемый `reason`.

### Пример ответа при потере lock

```json
{
  "error": "LOCK_LOST",
  "reason": "LOCK_TAKEN_OVER",
  "lockedBy": {
    "id": "user_42",
    "name": "Alex"
  }
}
```

Возможные значения `reason`:
- `LOCK_EXPIRED` — TTL истёк без heartbeat;
- `LOCK_TAKEN_OVER` — lock принудительно забран владельцем документа.

## 9.5 Принудительный захват lock

### Endpoint

`POST /api/v1/documents/:id/lock/takeover`

### Назначение

Принудительный перевод lock документа на текущего пользователя.

### Ограничение

Доступно только владельцу документа (`access_type = owner`). При попытке takeover без права backend возвращает `403 Forbidden`.

### Поведение

После успешного takeover предыдущий редактор при следующем heartbeat получает `423 Locked` с `reason: LOCK_TAKEN_OVER` и должен перейти в режим чтения.

## 9.5а Изменение права редактирования для shared-пользователя

### Endpoint

`PATCH /api/v1/files/:id/accesses/:accessId`

### Назначение

Изменение поля `can_edit` для уже выданного shared-доступа к файлу.

### Пример запроса

```json
{
  "can_edit": true
}
```

### Ограничение

Доступно только владельцу файла. Не применимо к `access_type = saved`.

## 9.6 Освобождение lock

### Endpoint

`DELETE /api/v1/documents/:id/lock`

### Назначение

Явное освобождение lock при штатном выходе пользователя из режима редактирования.

## 9.7 Сохранение документа

### Endpoint

`PUT /api/v1/documents/:id`

### Назначение

Сохранение обновлённого markdown-содержимого документа.

### Пример запроса

```json
{
  "content": "# Заголовок\n\nОбновлённый текст",
  "etag": "\"9b2cf535f27731c974343645a3985328\""
}
```

### Пример успешного ответа

```json
{
  "id": "file_789",
  "etag": "\"e1cbb0c3879af8347246f12c559a86b5\"",
  "updatedAt": "2026-05-16T07:10:00Z",
  "updatedBy": {
    "id": "user_15",
    "name": "Anna"
  }
}
```

## 9.8 Конфликт сохранения

### Условие

Если документ был изменён другим пользователем или текущая версия объекта уже отличается от той, что была открыта в редакторе, backend должен вернуть ошибку конфликта.

### Пример ответа

```json
{
  "error": "DOCUMENT_CONFLICT",
```

## 9.9 Получение доступных изображений

### Endpoint

`GET /api/v1/assets/images`

### Назначение

Получение списка изображений, доступных текущему пользователю для вставки в документ.

### Параметры

Допускается поддержка:
- поиска;
- пагинации;
- фильтрации по типу источника, например: мои, shared, все доступные;
- фильтрации по текущему контексту файловой системы.

## 9.10 Загрузка изображения из редактора

### Назначение

Редактор должен использовать существующий upload flow файловой подсистемы проекта через UI-обёртку.

### Требование

После завершения существующего сценария загрузки frontend должен получить данные, достаточные для немедленной вставки изображения в редактор.

## 10. Требования к frontend

### 10.1 Логика открытия

Frontend должен:
- определять, что файл относится к типу редактируемых markdown-документов (по `is_editable = true` и `editor_type = markdown`);
- вызывать endpoint получения документа;
- отображать состояние lock;
- если пользователь имеет право редактирования (`canEdit = true`) и документ свободен — инициировать получение lock;
- если пользователь не имеет права редактирования (`canEdit = false`) — открывать документ в режиме чтения без запроса lock;
- инициализировать Tiptap markdown-содержимым;
- открывать отдельный экран редактора (в режиме редактирования или чтения в зависимости от прав).

### 10.2 Логика сохранения

Frontend должен:
- извлекать markdown из редактора при сохранении;
- отправлять markdown и текущий `etag` на backend;
- после успешного ответа обновлять локальный `etag`;
- отображать актуальный статус сохранения.

### 10.3 Логика lock

Frontend должен:
- создавать lock перед началом редактирования (только при `canEdit = true`);
- отправлять heartbeat каждые 60 секунд;
- освобождать lock при штатном закрытии редактора;
- при получении `423` с `reason: LOCK_EXPIRED` — предложить пользователю заново получить lock;
- при получении `423` с `reason: LOCK_TAKEN_OVER` — перевести документ в режим чтения и показать сообщение с именем нового редактора;
- не отправлять heartbeat в режиме чтения.

### 10.4 Логика вставки изображения

Frontend должен:
- открывать picker изображений;
- запрашивать только доступные текущему пользователю изображения;
- вставлять выбранное изображение в текущую позицию курсора;
- при необходимости подставлять `alt` и `title` на основе имени файла.

### 10.5 Ошибки и уведомления

Frontend должен корректно обрабатывать:
- ошибку загрузки документа;
- ошибку сохранения;
- конфликт версии документа;
- ошибку получения или продления lock;
- недоступность выбранного изображения;
- отсутствие прав на редактирование;
- отсутствие прав на просмотр или вставку изображений.

## 11. Требования к backend

### 11.1 Общие требования

Backend должен:
- различать обычные файлы и markdown-документы;
- проверять права доступа к документу;
- проверять права доступа к изображениям, вставляемым в документ;
- работать с текущей файловой моделью и существующим S3-хранилищем.

### 11.2 Сохранение документа

Backend должен:
- принимать markdown-содержимое как текст;
- сохранять его в S3 по текущему `storageKey`;
- получать актуальный `etag` объекта из S3 после сохранения;
- обновлять метаданные файла;
- возвращать новую версию `etag` и данные о последнем изменении.

### 11.3 Защита от перезаписи

Backend должен:
- проверять наличие валидного lock для операции сохранения;
- проверять актуальность `etag` перед записью;
- не допускать тихой перезаписи документа поверх более новой версии;
- возвращать контролируемую ошибку конфликта при устаревшей версии.

### 11.4 Работа с изображениями

Backend должен:
- предоставлять endpoint получения списка доступных изображений;
- возвращать только те изображения, к которым у пользователя есть права доступа;
- отдавать стабильные URL для вставки в markdown-документ;
- не допускать использования временных URL в качестве постоянных ссылок внутри документа.

### 11.5 Работа с lock

Backend должен:
- хранить lock в отдельной таблице `document_locks` (поля: `file_id`, `user_id`, `expires_at`, `created_at`);
- создавать lock с TTL 5 минут;
- продлевать lock по heartbeat, сбрасывая `expires_at` на +5 минут;
- при heartbeat проверять, принадлежит ли lock текущему пользователю; если нет — возвращать `423` с машиночитаемым `reason: LOCK_EXPIRED` или `LOCK_TAKEN_OVER`;
- освобождать lock явным DELETE-запросом;
- поддерживать принудительный takeover только для владельца документа; после takeover прежний holder получит `LOCK_TAKEN_OVER` при следующем heartbeat;
- автоматически считать lock невалидным после истечения TTL без heartbeat (запись остаётся до следующей проверки или планировщика очистки).

## 12. Права доступа

### 12.1 Сводная таблица прав

| Право | Owner | Shared + can_edit | Shared (без can_edit) | Saved |
|---|---|---|---|---|
| Просмотр документа | ✅ | ✅ | ✅ | ✅ |
| Редактирование (lock + save) | ✅ | ✅ | ❌ | ❌ |
| Переименование документа | ✅ | ❌ | ❌ | ❌ |
| Удаление документа | ✅ | ❌ | ❌ | ❌ |
| Получение lock | ✅ | ✅ | ❌ | ❌ |
| Принудительный takeover lock | ✅ | ❌ | ❌ | ❌ |
| Изменение can_edit у доступов | ✅ | ❌ | ❌ | ❌ |
| Вставка изображений (`canInsertImages`) | ✅ | ✅ | ✅ | ✅ |
| Загрузка нового изображения | ✅ | ✅ | ✅ | ✅ |

### 12.2 Правила вычисления `canEdit`

- `access_type = owner` → `canEdit = true` всегда;
- `access_type = shared` и `can_edit = true` → `canEdit = true`;
- `access_type = shared` и `can_edit = false` → `canEdit = false`;
- `access_type = saved` → `canEdit = false`.

Изменение `can_edit` для shared-пользователя выполняется через `PATCH /api/v1/files/:id/accesses/:accessId` и доступно только владельцу.

### 12.3 `canInsertImages`

`capabilities.canInsertImages` означает право пользователя использовать функцию вставки изображений в документ.

Этот флаг не зависит от того, есть ли в системе доступные изображения в текущий момент.

Если право есть, но доступных изображений не найдено, picker должен открываться и показывать пустое состояние.

Если права нет, действие вставки изображений должно быть скрыто или недоступно в UI, а backend должен также запрещать соответствующие операции.

## 13. Ограничения и допущения MVP

Для MVP принимаются следующие ограничения:
- только один пользовательский режим редактирования, без отображения исходного Markdown;
- без совместного редактирования в реальном времени;
- без автоматического merge конфликтующих версий;
- без отдельной истории версий внутри интерфейса редактора;
- без поддержки сложной кастомной блочной структуры;
- без сохранения временных URL внутри документа.

## 14. Критерии готовности

Функциональность считается реализованной, если выполнены следующие условия:

1. Пользователь может создать новый файл `.md` через действие **«Создать заметку»** в файловом менеджере.
2. Новый файл сохраняется в S3 в формате Markdown.
3. Пользователь может открыть существующий `.md` файл не как скачиваемый объект, а как редактируемый документ.
4. Документ отображается в визуальном редакторе Tiptap.
5. Пользователь может изменить текст и форматирование и сохранить изменения.
6. После сохранения обновлённый Markdown записывается обратно в S3.
7. Пользователь может вставить в документ изображение из числа доступных ему файлов.
8. В документе сохраняются стабильные ссылки на изображения.
9. При активном редактировании другим пользователем отображается состояние блокировки документа.
10. Пользователь с соответствующим правом может принудительно забрать документ на редактирование.
11. При конфликте версий пользователь получает контролируемую ошибку, а данные не перезаписываются молча.
12. Все операции выполняются в рамках текущей системы прав доступа и существующей файловой инфраструктуры.

4. Реализовать `PATCH /api/v1/files/:id/accesses/:accessId` для изменения `can_edit`.
5. Реализовать backend-механизм стабильной выдачи приватных изображений: `GET /api/v1/files/:id/content` с проверкой прав и redirect на presigned URL.

9. Реализовать endpoint `GET /api/v1/assets/images` и UI picker изображений.
